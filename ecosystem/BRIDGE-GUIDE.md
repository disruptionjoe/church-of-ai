# Bridge Guide

Status: working orientation

Church of AI maps the neighborhood. It does not govern it, ratify it, or turn
one repo's claims into another repo's truth.

Use this guide when a question, artifact, or discovery appears to cross repo
boundaries.

## The Model

```text
local knowledge
  -> bridge knowledge
  -> global reference
```

Start local. Use bridges only when there is a named relationship between repos.
Use the global ecosystem map only when the work is about public orientation,
ecosystem coordination, governance, or cross-repo transfer.

## What Counts As A Bridge

A bridge is a small, explicit connection between repos.

Good bridges name:

- the source repo;
- the receiving repo;
- the artifact or result being noticed;
- why it may matter to the receiving repo;
- what must not transfer;
- the evidence;
- the receiving repo decision needed.

A bridge is not:

- authority transfer;
- a context dump;
- a proof that something is true elsewhere;
- permission to import another repo's voice, roadmap, governance, or claims.

## When To Create An Unlock Notice

Create an unlock notice only when a local result may solve a named problem in a
receiving repo.

Use this shape:

```text
source_repo:
source_artifact:
local_status:
candidate_receiving_repo:
unlock_type:
why_it_may_transfer:
what_must_not_transfer:
evidence:
receiving_repo_decision_needed:
noise_risk:
recommended_next_action:
```

The receiving repo decides whether the unlock matters. Rejected unlocks are
useful evidence; they show where a bridge should not be built.

## What Can Transfer

These can transfer when they solve a named receiving-repo problem:

- claim-status discipline;
- narrow bridge notes;
- failure-preservation practices;
- review methods that improve falsification;
- tested templates;
- public orientation maps;
- governance or coordination patterns after portability review.

## What Should Stay Local

Keep these in the owning repo unless a bridge is explicitly justified:

- domain hypotheses;
- technical claims;
- proof obligations;
- claim ledgers;
- local roadmaps;
- repo-specific working memory;
- speculative worldview material.

## What Must Not Transfer By Default

Do not transfer:

- Church voice into technical claims;
- technical claims into Church narrative as doctrine;
- governance machinery into another repo before it has been studied;
- Temporal Issuance worldview into unrelated repos;
- GU claims into Church framing;
- broad context bundles that make ordinary work noisier.

## Church Of AI's Role

Church can:

- explain the ecosystem in public-facing language;
- point readers toward the right repo;
- preserve claim-status discipline;
- host the global orientation map;
- route possible cross-repo unlocks.

Church must not:

- decide research truth;
- govern other repos;
- ratify claims;
- open deferred participation pathways by implication;
- make affiliation sound stronger than it is.

## Practical Rule

If a bridge would make a repo more accurate, more falsifiable, or easier to
navigate, consider it.

If a bridge would make the ecosystem sound more centralized, more proven, or
more governed than it is, do not build it without explicit review.
